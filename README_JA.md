# Flowbase Admin Dashboard（フロントエンド構造デモ）

本プロジェクトは、**マルチテナント型 B2B SaaS を想定したフロントエンド構造**を示すためのデモアプリケーションです。

インターンシップで経験した、HRMS およびタスク管理プロダクトにおける
**ロール設計・組織（Workspace）切り替え・ルーティング制御・UI 構造設計**といった実務内容を、
**個人プロジェクトとして再構築・整理**することを目的に制作しています。

実際の業務で関わった設計判断やユーザーフローを、
**プロプライエタリなコードを含まない形**で再現し、
バックエンドを実装せず **モックデータのみ**を用いることで、
フロントエンドにおける構造設計・状態管理・UI 分岐の考え方にフォーカスしています。

---

## デモ（Live Demo）

[https://flowbase-five.vercel.app]

以下のデモログインが利用できます：

- **Admin ダッシュボード**
- **Member ダッシュボード**

**ユーザー名やパスワードの入力は不要**です。

## プロジェクトの目的

このプロジェクトの目的は、見た目の作り込みではなく、

- 状態管理を前提とした画面構造
- B2B SaaS における典型的なユーザーフロー
- フロントエンド側での責務分離と設計判断

を **コードと UI の両面から説明できる形で示すこと**です。

---

## 主な機能・構成

### ① モックログイン（Admin / Member）

- 実際の認証機能は実装していません
- あらかじめ定義したモックユーザーを用いてログインを再現しています

**対応ロール**

- **Admin**：複数の組織に所属、管理者権限あり
- **Member**：単一組織のみ、管理者権限なし

※ 認証を省略することで、
フロントエンドの構造・設計に集中しています。

---

### ② マルチテナント対応（Org Picker）

- ユーザーは 1 つまたは複数の組織（Workspace）に所属可能
- 複数組織を持つユーザーは、ログイン後に組織選択画面へ遷移します
- 選択した組織はアプリ全体のコンテキストとして保持されます

**ログイン後のフロー**

```
ログイン
→（複数組織の場合）Org Picker
→ ダッシュボード
```

---

### ③ ロール別サイドバーナビゲーション

- Sidebar はログインユーザーのロールに応じて内容が変化します
- Admin のみが閲覧できる管理系メニューを明確に分離しています
- UI 表示だけでなく、後述のルートガードと組み合わせて制御しています

---

### ④ ダッシュボード構成について（Admin / Member 分離）

本プロジェクトでは、管理者（Admin）と一般ユーザー（Member）で  
**ダッシュボードの内容を明確に分離**しています。

レイアウト（グリッド構造やカード配置）は共通としつつ、  
表示する情報はそれぞれのロールに応じて切り替えることで、  
**組織全体を管理する視点**と **個人として利用する視点**の違いを表現しています。

- **Admin Dashboard**

  - Org Health Summary
  - Pending Approvals
  - Onboarding & Compliance
  - Subscription & Billing
  - Recent Activity（組織全体）
  - Quick Admin Actions

- **Member Dashboard**
  - My Tasks
  - My Onboarding Status
  - Recent Activity（個人）
  - My Profile
  - Documents & Requests
  - Help & Resources

このように、UI の見た目だけでなく  
**「誰が・何を見るべきか」**という責務を  
フロントエンド構造として分離することで、  
実務に近いロール設計とユーザー体験を再現しています。

---

### ⑤ 管理者専用ルートのアクセス制御（補足）

- `/admin/*` 配下は管理者専用ルートとして実装
- 非管理者が URL を直接入力した場合でも  
  Not Authorized ページへリダイレクト
- UI 表示に依存しない、ページ単位でのガードを採用

---

### ⑥ グローバルレイアウトとコンテキスト表示

- Header には現在選択中の組織名を表示
- ユーザーが「どの組織の文脈で操作しているか」を常に把握できる UI 構成
- 状態管理（Zustand）を通じて全体に反映しています

## 技術スタック

- **Framework**：Next.js（App Router）
- **UI**：React / Tailwind CSS
- **状態管理**：Zustand
- **ルーティング**：Next.js 標準ルーティング
- **データ**：モックデータ（バックエンド未使用）

## ルーティング構成

```
/               エントリーポイント（状態に応じてリダイレクト）
/login          モックログイン画面
/org-picker     組織選択画面（複数組織ユーザー）
/dashboard      メインダッシュボード
/admin/users    管理者専用ページ
/not-authorized 権限なし表示
```

## 設計上の判断について

### なぜ認証をモックにしているか

本プロジェクトでは、以下を重視しています。

- フロントエンドの構造設計
- ルーティングと状態の関係性
- 実務に近い UI 分岐の再現

そのため、実際の認証処理は意図的に省略し、
**ロール・組織・状態遷移の設計**にフォーカスしています。

---

### なぜページ単位でアクセス制御しているか

- UI 非表示だけではセキュリティ上不十分
- URL 直打ちを想定した制御が必要
- ロジックを明示的にコードで表現したい

という理由から、各管理者ページで
**明確なアクセスガード処理**を行っています。
